# خطة تطوير الـ API للمشروع الطبي (FastAPI + PostgreSQL)

توضح هذه الوثيقة محطات التطوير (Milestones) مقسمة بشكل تدريجي لبناء خادم التطبيقات وقاعدة البيانات، وربطها بالواجهة الأمامية للموقع.

---

## 🏁 المحطة 1: إعداد بيئة العمل وهيكل المشروع الأساسي
*الهدف: إنشاء البيئة البرمجية وضمان استقرار الاتصال بقاعدة البيانات.*

- [x] إنشاء بيئة افتراضية لـ Python 3.10+ في المجلد `backend/`.
- [x] إعداد ملف `requirements.txt` وتثبيت المكتبات الأساسية:
  - `fastapi`, `uvicorn` (خادم ويب وإطار العمل)
  - `sqlalchemy`, `psycopg2-binary` (الاتصال بالـ PostgreSQL و ORM)
  - `alembic` (إدارة تعديلات وتحديثات قاعدة البيانات)
  - `pydantic` (التحقق من صحة البيانات الواردة)
  - `python-jose`, `passlib[bcrypt]` (تأمين الهوية وتشفير كلمات المرور)
  - `python-dotenv` (لقراءة المتغيرات البيئية)
- [x] تصميم الهيكل البرمجي للمجلدات:
  - `backend/app/core/` (إعدادات النظام والأمن والاتصال بقاعدة البيانات)
  - `backend/app/models/` (تعريف الجداول في قاعدة البيانات)
  - `backend/app/schemas/` (نماذج Pydantic للتحقق والإرجاع)
  - `backend/app/api/` (وحدات التحكم ونقاط الاتصال البرمجية)
  - `backend/app/crud/` (العمليات المباشرة لقراءة وكتابة البيانات)
- [x] إنشاء ملف الإعدادات البيئية `.env` وتكوين جملة الاتصال بـ PostgreSQL و مفتاح التشفير JWT Secret Key.
- [x] تهيئة Alembic للمشروع للبدء في تتبع التعديلات.

## 🗄️ المحطة 2: تصميم نماذج قاعدة البيانات وهجرة الجداول (Migrations)
*الهدف: بناء وتجهيز قاعدة البيانات لاستقبال البيانات وتخزينها.*

- [x] كتابة نماذج قاعدة البيانات (SQLAlchemy Models):
  - [x] جدول المستخدمين والصلاحيات (`User`)
  - [x] جدول محتوى الصفحة الرئيسية (`CMSHome`)
  - [x] جدول الخدمات والأسعار (`CMSService`)
  - [x] جدول الأطباء ومواعيدهم (`CMSDoctor`)
  - [x] جدول الباقات الصحية (`CMSPackage`)
  - [x] جداول المدونة وتصنيفاتها (`BlogPost` و `BlogCategory`)
  - [x] جدول الحجوزات الطبية (`Booking`)
  - [x] جدول رسائل اتصل بنا (`ContactMessage`)
- [x] تكوين العلاقات البرمجية (Foreign Keys & Relationships) بين الجداول (مثل ربط المقال بالتصنيف).
- [x] توليد ملف الهجرة الأول باستخدام Alembic وتطبيق الجداول في قاعدة بيانات PostgreSQL (`alembic upgrade head`).
- [x] إنشاء وتشغيل سكربت لتعبئة قاعدة البيانات ببيانات تجريبية أولية (Seed Data) مأخوذة من mock data الخاص بالواجهة الأمامية لضمان تجربة الموقع مباشرة.

## 🔒 المحطة 3: نظام المصادقة والصلاحيات (Auth & RBAC)
*الهدف: حماية لوحة التحكم وتقييد الوصول حسب الرتب والأدوار للمستخدمين.*

- [ ] إعداد دوال تشفير كلمات المرور باستخدام خوارزمية `bcrypt`.
- [ ] برمجة دوال إصدار توكن JWT والتحقق منه مع تاريخ الانتهاء.
- [ ] تطوير نقطة الاتصال `POST /api/auth/login` للتحقق من هوية المشرفين والأطباء وإرجاع التوكن.
- [ ] تطوير وظيفة وسيطة (Dependency Injection) للتحقق من هوية المستخدم الحالي عبر التوكن (`get_current_user`).
- [ ] تطوير نظام التحقق من الصلاحيات والأدوار (Role-Based Access Control) لتقييد الوصول لبعض المسارات حسب المذكور في الـ PRD (مثال: تقييد تأكيد الحجوزات لموظفي الاستقبال والمدراء فقط).
- [ ] تطوير نقطة الاتصال `GET /api/auth/me` لجلب بيانات المستخدم المسجل حالياً وصلاحياته الفعالة.

## 🖥️ المحطة 5: لوحة التحكم لإدارة الحجوزات والرسائل (Admin CMS API)
*الهدف: السماح لموظفي الاستقبال والمدراء بإدارة العمليات الحيوية للمركز.*

- [ ] تطوير نقاط اتصال إدارة الحجوزات (للطاقم الطبي والإداري):
  - [ ] `GET /api/bookings` (تتطلب صلاحية `bookings.view` لعرض الحجوزات مع تصفية متقدمة بالتاريخ والحالة والطبيب).
  - [ ] `PUT /api/bookings/{booking_id}/status` (تتطلب صلاحية `bookings.confirm` لتأكيد الحجز أو إلغائه).
  - [ ] `DELETE /api/bookings/{booking_id}` (تتطلب صلاحية `bookings.delete` لحذف سجل الحجز).
- [ ] تطوير نقاط اتصال إدارة الرسائل:
  - [ ] `GET /api/contact` (تتطلب صلاحية `messages.view` لاستعراض الرسائل الواردة).
  - [ ] `PUT /api/contact/{message_id}/status` (تتطلب صلاحية `messages.reply` لتحديث حالة الرسالة إلى مقروءة أو تمت الإجابة عليها).
  - [ ] `DELETE /api/contact/{message_id}` (تتطلب صلاحية `messages.delete`).
